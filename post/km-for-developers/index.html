Knowledge Management for Developers

November 27, 2022
Software

As I understand it, Tiago Forte is the person credited with the term ‘second brain’, in reference to his solutions to the problem of personal knowledge management. I say ‘solutions’ because he goes on to recommend using a stack of software applications for essentially one thing - email, to do list, task managers and some notebook application. In a similar vein, the ‘second brain’ concept had spawned a number of related ‘solutions’, such as Zettelkasten, PARA, Roam Research, Notion.so/, Obsidian, knowledge graphs, etc.

Notion.so/ is okay for experimenting with structures and layouts of Web sites, and maybe as a shared wiki for non-sensitive information. Obsidian is pretty good as a text editor - I actually like it. In the context of personal knowledge management, I’d argue that all the above are largely solutions in need of a problem, and that conventional methods of organising information - that is, simply making notes and putting them in directories and subdirectories, without any definite prescribed software or method - will always be far more effective. And it can be done with just an editor (preferably a lightweight code editor) that has a few generic features. I’m also arguing that this could more accurately be called a ‘second brain’ because it’s more aligned to how the human mind works.

My first reservation with the fancy ‘second brain’ solutions - insofar as they recommend adding a note for every idea and thought, is they’re unsustainable, producing a disorganised hoard of thoughts, ideas and facts absent of context. Can such a collection of things really be synthesised into knowledge, or expertise?

The Use Case

We maintain about 150 services and integrations. Each of these will have its own process for development, testing and deployment. Obviously this is too much information to remember, and we need to create our own documentation for this. We need something to document:

  • Code that can be repurposed.
  • Environments and their configurations.
  • URIs for the services and integrations.
  • Deployment process.
  • Testing process
  • Mappings between services and integrations.
  • Which databases the services and integrations use
  • Deployment pipelines
  • Change requests
  • Code samples and templates that can be repurposed
  • Configuration secrets - these can’t be uploaded to GitHub.

Most of my colleagues share this information in Markdown format, which is versatile enough to paste into GitHub repos, static site generators and a range of other software capable of rendering and converting the content into other formats. Some of us are also using Jupyter Labs or Azure Data Studio, which are essentially Markdown editors capable of running embedded SQL - this is a handy method of storing code and documentation in the same file.

Why Information Should be Hierarchical

That information should be organised hierarchically seems an axiom. When it comes to software that presents a large number of items in the UI, developers will invariably opt for methods that organise those items into a tree structure, rather than, say, a graph. Any technical manual will have the information organised into sections and sub-sections. And there must be a reason the UNIX file system was designed that way.

I think the reason is it’s consistent with how the human mind works and it’s that particular method of organising that requires the least effort. We have a tendency to organise complex data that way, even if just mentally, and to form abstractions and generalisations. With hierarchical data, our minds need only a vague conceptual model in order to know where to find something specific. Anyone could, for example, learn the names of six (or is it seven) seven root directories, and make a pretty decent guess where any given piece of information about a Linux system could be found. The more that person does that, the more detailed the conceptual model of the system becomes. And it’s productive, because that method can produce expertise in a way that a disorganised aggregate of facts couldn’t.

What I’m Using

After trying a few such solutions over the last year or so, from Zettelkasten to Notion.so/, it turned out I only needed an editor that does the following: Open a directory of files. Find keywords and phrases within that directory of files. Process Markdown (ideally with syntax highlighting. Have a decent, customisable UI that makes it easy to scan through large amounts of text. Be portable - our solution shouldn’t be tied to any specific software application, and the same directory of files should be usable in other editors with the same features. It should be local, and shouldn’t require the uploading of sensitive information to third-party servers.

FOAM and Visual Studio Code

Currently I’m using the FOAM Visual Studio Code extension, and it adds a number of useful features to the editor: It enables us to use tags to mark and retrieve files related to a subject area, documents can be linked, and we can search for keywords and phrases within a large number of files. FOAM has scaled up far better than the PARA and Zettel methods, and can be used with a variety of other code editors. Another huge advantage of FOAM is it will run alongside any number of extensions in VS Code that provide additional features such as Markdown syntax highlighting, the encryption of sensitive files and strings, and, of course, PowerShell commands could be run against the current directory.

It wasn’t perfect, though. Until recently, there were a couple of annoying quirks with my installation of Visual Studio that required some fixing.

Tips for using FOAM in VS Code

To get rid of the annoying update notifications that appear every time VS Code is started: “update.mode”: “none”

VS Code might automatically indent wrapped text/Markdown, making the content awkward to read. Somewhere in the results, there is a ‘Editor: Wrapping Indent’, and next to it, ‘(Also modified in Workspace)’. Click on the Workspace link to access the setting and disable the indent. editor.wrappingIndent If we’re using the directory in more than one installation of VS Code, it’s a good idea to create a workspace file and add the above line to it.

It’s possible to override the configuration and UI styling, in a JSON file, just for the currently loaded project. See my configuration file for an example. ‘File’ -> ‘Save Workspace As…’. This will create a JSON file in the root of the project directory, with the ‘.code-workspace’ extension, where all kinds of configuration fields can be added and modified.
